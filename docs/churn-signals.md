# Churn Signals

Two layers feed customer churn / health scores: structured (typed) and AI-extracted (JSONB).

## 1. Structured signals — `churn_signals` table

Quantitative, time-stamped. CHECK constraint enforces 5 allowed types:

| `signal_type` | What it represents | Typical scale |
|---|---|---|
| `usage` | Product usage / activity score | 0.0 – 1.0 (lower = worse) |
| `login` | Login frequency or session count | 0.0 – 1.0 ratio |
| `nps` | Net Promoter Score response | -100 to 100, often 0–10 |
| `support` | Support ticket volume | integer |
| `csat` | CSAT survey response | 0.0 – 1.0 |

Each row is one observation: `(customer_id, signal_type, value, recorded_at)`. Multiple rows per customer build the time series.

Loaded via the `churn_signals` upload template or piped from product analytics.

## 2. AI-extracted signals — `csm_notes.ai_extracted_signals` (JSONB)

Generated by Claude when a `csm_notes` upload runs (`extract_csm_signals` in `signal_extractor.py`):

| Key | Type | Meaning |
|---|---|---|
| `competitor_mention` | `string \| null` | Competitor named in the conversation |
| `budget_pressure` | `boolean` | Customer flagged budget cuts / cost concerns |
| `sponsor_change` | `{departed, incoming} \| null` | Champion/decision-maker leaving or replaced |
| `expansion_interest` | `boolean` | Upsell/expansion opportunity surfaced |
| `satisfaction_level` | `'positive' \| 'neutral' \| 'negative'` | Overall tone |
| `key_concerns` | `string[]` | Specific concerns raised |
| `commitments_made` | `string[]` | Next steps / commitments |
| `churn_risk_signals` | `string[]` | Direct quotes flagging churn risk |

## 3. Aggregate scores (per customer)

Stored on the `customers` row; written by the rescoring pipeline:

| Field | Range | Meaning |
|---|---|---|
| `latest_churn_score` | 0–100 | Higher = more likely to churn |
| `latest_health_score` | 0–100 | Higher = healthier |
| `latest_revenue_at_risk` | USD | Estimated ARR at risk |
| `scores_last_updated_at` | timestamp | When rescoring last ran |

## How they combine

Both layers feed the Claude rescoring prompt (`_rescore_customer` in `scoring.py`):

```
Inputs to Claude:
  customer profile (segment, arr, renewal_date, current scores)
  csm_notes.ai_extracted_signals  ← AI signals (most recent)
  churn_signals (recent rows)     ← structured signals
  RAG context from csm_notes embeddings

Output:
  churn_score, health_score, revenue_at_risk
  + reasoning (plain English)
  + contributing_factors (weighted)
```

## When scores update

| Trigger | Source |
|---|---|
| `upload` | New `csm_notes` upload kicks off rescoring of affected customers |
| `query` | Manual re-run from a query in the UI |
| `scheduled` | (Future) periodic refresh |
| `initial` | First-ever score on customer creation |

The `trigger_type` field on `customer_scores` and `ai_analysis_runs` records which path produced the score.
